# Phase 4: Building respondA (Web UI)

## Goal
Build the unified dashboard (`respondA`) for analysts to view real-time alerts generated by `alertA` and explore the underlying raw logs in BigQuery. The UI will connect directly to Firestore for real-time state management.

## Prerequisites
1. Phases 1-3 are complete.
2. Firestore contains populated `alerts` documents.
3. BigQuery contains the enriched `events` table.

## Steps

### 1. Project Setup
- Navigate to `services/respondA/`.
- Initialize a frontend application (e.g., using Vite with React/TypeScript or Vue).
- Install dependencies: `firebase` (client SDK), `@google-cloud/bigquery` (if routing queries through a backend API), and UI component libraries (e.g., Material UI or Tailwind).

### 2. Frontend Architecture & Types
- Ensure the frontend imports or matches the alert schema defined in `shared/models/alert`.
- Configure the Firebase Client SDK using a Firebase Web App configuration mapped to the GCP project.

### 3. Core Feature 1: The Alert Dashboard (Real-time Firestore)
- Create an "Alerts Queue" view.
- Use the Firebase SDK's `onSnapshot` method to listen to the `alerts` collection where `status == "OPEN"`.
- This ensures the UI updates instantly the moment `alertA` creates a new document, without polling.
- Implement buttons to transition alert status in Firestore (e.g., "Acknowledge", "Resolve").

### 4. Core Feature 2: Log Exploration (BigQuery Search)
- Since BigQuery cannot be queried directly from the browser safely without a proxy, create a simple backend route (either within a Next.js/Nuxt server route, or an additional FastAPI endpoint on `alertA` or `ingestA`).
- Create a "Log Search" view in the UI.
- Provide inputs for Time Range, Source, and Free Text search.
- Translate these inputs into parameterized BigQuery SQL (e.g., `SELECT * FROM defenda_data_lake.events WHERE...`) and display the resulting rows.

### 5. Containerization & Deployment
- Write a `Dockerfile` for `respondA`.
- If using a static build (React/Vite), use a multi-stage build that compiles the static assets and serves them via Nginx.
- Build and push the image to Artifact Registry.
- Update the Terraform configuration to deploy the `respondA` image to Cloud Run. Ensure this specific Cloud Run service allows public/unauthenticated access (or is secured via Google Identity-Aware Proxy for enterprise access).

## Success Criteria
- Navigating to the `respondA` Cloud Run URL successfully loads the UI.
- Creating a dummy alert in Firestore via the GCP Console instantly appears in the UI without a page refresh.
- Clicking "Resolve" on an alert in the UI successfully updates the Firestore document status.
- Analysts can execute a basic log search from the UI that returns actual rows from the BigQuery `events` table.
