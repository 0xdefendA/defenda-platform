# Architecture Design: defenda-platform

## Overview
`defenda-platform` is a unified, cloud-native security platform deployed entirely on Google Cloud Platform (GCP). It represents the evolution of the `defenda-data-lake` (AWS) and `alertA` (MongoDB/Docker) projects into a cohesive, serverless monorepo. 

The platform provides a scalable data lake for unstructured JSON log ingestion, a high-speed alerting engine for threat detection, and a real-time user interface for analyst response.

## Core Services & Naming Convention
The platform consists of three primary services following the "A" motif:

1. **`ingestA` (Data Lake Engine)**: Handles receiving, normalizing, and enriching unstructured JSON logs.
2. **`alertA` (Alerting Engine)**: Continuously queries the data lake against defined signatures/rules to identify anomalies.
3. **`respondA` (User Interface)**: The centralized dashboard for security analysts to view, triage, and resolve alerts generated by `alertA`.

## Architecture & GCP Mapping

To achieve a fully serverless, highly scalable, and zero-maintenance infrastructure, the legacy AWS/Docker components are mapped to the following GCP services:

| Legacy Component | New GCP Component | Purpose |
| :--- | :--- | :--- |
| AWS S3 | **Cloud Storage (GCS)** | Raw log archival and long-term storage. |
| AWS Kinesis Firehose | **Pub/Sub** | High-throughput message queuing and routing. |
| AWS Lambda | **Cloud Run** | Containerized, serverless execution for `ingestA` and `alertA`. |
| AWS Athena | **BigQuery** | Serverless data warehouse with native JSON support for log queries. |
| MongoDB (Docker) | **Firestore** | Real-time, managed NoSQL document database for alert state. |
| AWS EventBridge | **Cloud Scheduler** | Triggering periodic queries in `alertA`. |

### Database Strategy (OLAP vs OLTP)
The platform intentionally separates its database workloads to optimize for cost and performance:
- **BigQuery (OLAP)**: The source of truth for all raw and enriched logs. Optimized for scanning massive datasets, time-based partitioning, and deep JSON extraction.
- **Firestore (OLTP)**: The source of truth for alert states, user configurations, and UI data. Optimized for fast, single-document point lookups, high-frequency updates, and real-time UI synchronization.

## Data Flow
1. **Ingestion**: Log producers send raw JSON payloads to a **Pub/Sub topic**.
2. **Processing (`ingestA`)**: A **Cloud Run** service is triggered by the Pub/Sub push. It executes Python normalization/enrichment plugins on the payload and streams the structured/enriched JSON into a **BigQuery** table.
3. **Detection (`alertA`)**: **Cloud Scheduler** triggers a **Cloud Run** service every minute. `alertA` executes a suite of SQL queries against **BigQuery**. If an anomaly is found, it evaluates the rule against the current state in **Firestore**. If a new alert is warranted, it creates a document in **Firestore**.
4. **Response (`respondA`)**: The web UI maintains a real-time listener connection directly to **Firestore**. When `alertA` creates a new document, it instantly appears on the analyst's dashboard. Analysts interact with the UI to update the alert state (e.g., status: "resolved") in **Firestore**.

## Monorepo Structure

The project is structured as a monorepo to ensure unified deployments and shared schemas.

```text
defenda-platform/
│
├── cicd/                # Infrastructure as Code (GCP)
│   ├── main.tf               
│   ├── variables.tf
│   └── outputs.tf
│
├── services/
│   ├── ingestA/              # Cloud Run service (Python/FastAPI)
│   │   ├── src/
│   │   ├── plugins/          # Ported from defenda-data-lake
│   │   └── Dockerfile
│   │
│   ├── alertA/               # Cloud Run service (Python)
│   │   ├── src/
│   │   ├── rules/            # YAML/JSON alert definitions
│   │   └── Dockerfile
│   │
│   └── respondA/             # Web UI (TypeScript/React or Vue)
│       ├── src/              
│       └── Dockerfile
│
├── shared/                   # Shared types, constants, or utility functions
│   └── models/               # E.g., Alert schema used by both alertA and respondA
│
├── docker-compose.yml        # Local development (GCP emulators)
├── .github/workflows/        # CI/CD pipelines
└── README.md
```

## Advantages of this Architecture
- **Native JSON Querying**: BigQuery's native `JSON` column type eliminates the need for complex string-extraction functions (e.g., `json_extract_scalar`). Queries use clean dot notation (`WHERE STRING(details.eventname) = 'ConsoleLogin'`).
- **Zero Partition Management**: BigQuery handles time-based partitioning automatically upon ingestion. The `generate_partitions.py` lambda from the AWS architecture is entirely eliminated.
- **Real-time UI**: Firestore's native client SDKs allow the UI (`respondA`) to update instantly without polling.
- **Portability**: Transitioning from Lambda zip files to Cloud Run standard Docker containers makes local testing trivial and prevents vendor lock-in at the compute layer.

CICD pipeline

- Accessible via ./legacy/cicd
- Concept of operations: 
    - cicd/prod is the main directory, the module system is to assist with any dev/non prod projects (we may add dev later)
    - Goal is to establish a cloud build trigger using cloudbuild-apply.yaml on push to a branch (main)
    - and not expose any project ids, bucket names, etc in source code
    - We do this by storing the terraform.tvfars as a GCP secret (tfvars) that references our project/buckets like the terraform state bucket, etc at the time of run in cloudbuild. 
- Once the base services are roughed into terraform
- open a shell in cicd/prod
- render the backend.tf file inert (we don't have a bucket yet) by renaming to backend.tf.inert
- run ```terraform init``` to initialize terraform and providers.
- run ```terraform plan -target=module.gcp_project_setup``` to check the bootstrap build plan
- run ```terraform apply -target=module.gcp_project_setup``` to bootstrap the project and build pipeline
- prompt the user to complete the terraform init -force-copy to enable the backend.tf/bucket.
